How To Use This Document
This playbook is written to be executed by an autonomous coding agent, one phase at a time, with a human reviewing the completion report before the next phase is unlocked. Each phase includes: (1) the specific Ahrefs issue categories it resolves, (2) a task list, (3) a definition of done, and (4) a mandatory completion report. The agent should not skip ahead, batch phases together, or 'fix' issues not named in the current phase.
Reality check on "100 health score, zero issues": Ahrefs Site Audit's Health Score is a weighted composite across dozens of checks, and a small number of them depend on things outside application code — third-party embeds, external backlinks pointing at old URLs, DNS/CDN/TLS configuration, or issues Ahrefs flags on pages the business intentionally excludes from indexing. The realistic, honestly-achievable target is: 0 Errors, 0 Warnings on everything within the codebase's control, and a documented, reviewed justification for any Notice-level item that cannot be fixed in code. Treat "100" as the aspirational ceiling, not a pass/fail gate the agent should silently force by, e.g., deleting pages or blocking Ahrefs' crawler.

Global Rules
Work one phase at a time. Do not start Phase N+1 until Phase N's completion report is reviewed and accepted.
Validate before proceeding — re-run the relevant tool (Ahrefs, Screaming Frog, GSC, Lighthouse) after each phase, not just at the end.
No unrelated changes — a phase touching hreflang tags must not also touch canonical tags, copy, or styling.
Back up / branch before modifying — every phase's changes live in their own commit or PR, never mixed with another phase.
Never silently suppress an issue (noindex-ing a page, blocking a crawler, disabling a check) to make a health score look better. Suppression is only acceptable when it is the correct SEO fix and must be called out explicitly in the completion report.
When a fix requires a business decision (e.g., which URL should be canonical, whether a locale-root page should exist at all), stop and flag for human review rather than guessing.

Finding 1 — Duplicate hreflang annotations (190/190 groups, 100%)
Every hreflang group emits each alternate-language link twice in a row (5 total link entries where 2–3 unique URLs are expected). Example (group id 5927611169620563282):
en URL — emitted twice
es URL — emitted twice
unprefixed URL — emitted once, returning 307 (see Finding 2)
This is what triggers both "One page is linked for more than one language" and "More than one page is linked for the same language" on every affected URL.
Fix: locate the component/partial that renders <link rel="alternate" hreflang="..."> tags and remove the duplicate emission — it is almost certainly looping over the locale list twice (e.g., once for the tags and once for an og:locale:alternate or JSON-LD block using the same loop, or a partial being included twice in <head>).
Finding 2 — Unprefixed root URL included as a hreflang alternate, resolving 307 (190/190 groups, 100%)
Every group's hreflang set includes a third URL with no /en/ or /es/ prefix (e.g. www.kinexisdigital.com/industries/financial-services/insurance), and that URL returns HTTP 307 rather than 200. This is what triggers "Some pages are not returning 200 status code" and contributes to "Incorrect value" on every group.
This unprefixed URL is almost certainly a locale-detection redirect (mid-dleware sends the visitor to /en/ or /es/ based on browser language) that is being mistakenly included in the hreflang tag set instead of being excluded from it.
Fix — requires one decision from the business/SEO owner, flag for review:
Option A (usually correct): remove the unprefixed URL from hreflang annotations entirely. Hreflang should only ever point at pages that return 200 directly — never at a redirector. The /en/ and /es/ URLs remain reciprocal alternates of each other.
Option B: if the unprefixed URL is meant to serve as the x-default target for visitors with no matching locale, it must itself return 200 with real content (not a redirect), and be explicitly tagged hreflang="x-default" rather than being silently folded into the en/es set.
Note: this export does not capture the literal hreflang language-code attribute values (only URLs and their status codes), so the agent must inspect actual page source to confirm current x-default usage before choosing between Option A and B.
Tasks
1. Inspect the hreflang-rendering template/component; identify why each alternate is emitted twice; remove the duplicate render path.
2. Get a decision (Option A or B above) on the unprefixed URL from the SEO/business owner; implement accordingly.
3. Confirm reciprocity: /en/X must list /es/X as an alternate and vice versa, exactly once each.
4. Re-crawl (Ahrefs re-crawl or fresh export) and confirm all 190 groups now show 2 unique links (en+es) or 3 with a correctly-implemented 200 x-default — zero groups with duplicate or non-200 entries.
5. Spot-check the two non-indexable pages found in the export (/en/lead-magnet, /es/lead-magnet) — confirm whether they should be indexable or should be excluded from hreflang entirely.
